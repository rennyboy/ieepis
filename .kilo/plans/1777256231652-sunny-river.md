# Plan: Unify Users and Employees - Prevent Duplication

## Overview
The codebase has separate `users` and `employees` tables with no explicit foreign key relationship between them. Both tables have `email` and `school_id` fields, allowing the same person to exist independently in both tables. This plan establishes a formal 1:1 relationship by adding `user_id` FK to the `employees` table.

## Current State Analysis

### Schema
- **`users`** (id, name, email, password, school_id, approval_status, division, division_id)
  - `id` is PK
  - Has role-based access (Spatie)
  - Used for authentication

- **`employees`** (id, school_id, employee_number, first_name, last_name, email, position, ...)
  - `id` is PK  
  - No link to users table
  - Used for equipment assignments, tickets, documents

### Problem
- Same person can exist as both user AND employee
- No referential integrity between the tables
- Seeding creates matching email addresses but no FK link
- Risk of data inconsistency and duplication

## Solution: Add user_id Foreign Key to Employees

### Step 1: Create Migration - Add user_id to employees (nullable)
```php
$table->foreignId('user_id')->nullable()->constrained()->cascadeOnDelete();
```
- Nullable to allow existing employees without user accounts
- Cascade on delete: if user is deleted, employee record is also deleted
- FK constraint to users.id

### Step 2: Create Migration - Backfill Existing Relationships
Match employees to users by:
```sql
UPDATE employees e 
SET user_id = u.id 
FROM users u 
WHERE e.email = u.email 
  AND e.school_id = u.school_id
```
- Uses composite match on email + school_id
- Only where exact matches exist

### Step 3: Audit Unlinked Records
After backfill, identify:
- **Orphaned employees**: Employees with email+school that don't match any user
- **Users without employees**: User accounts with no corresponding employee record
- **Duplicate matches**: Multiple employees matching same user (email conflict)

### Step 4: Add Unique Constraint
Once backfill is verified:
```php
$table->unique('user_id');  // 1:1 relationship
```
Prevents multiple employees linking to the same user.

### Step 5: Update Employee Model
```php
public function user(): BelongsTo
{
    return $this->belongsTo(User::class);
}
```

### Step 6: Update Seeders
- DatabaseSeeder: Set user_id when creating employee
- DapitanSchoolsSeeder: Set user_id when creating ICT coordinator employee
- Use `->create()` with user_id instead of separate operations

### Step 7: Update Application Logic
- Where user+employee are created together (e.g., registration), set user_id
- Validate: Cannot create employee with user_id that's already taken (after unique constraint)
- Update queries that join users/employees to use FK instead of email matching

## Data Integrity Checks

### Pre-migration check
```sql
-- Check for email+school duplicates across tables
SELECT email, school_id, COUNT(*) 
FROM users 
GROUP BY email, school_id 
HAVING COUNT(*) > 1;

SELECT email, school_id, COUNT(*) 
FROM employees 
GROUP BY email, school_id 
HAVING COUNT(*) > 1;
```

### Post-backfill verification
```sql
-- Employees without user match
SELECT * FROM employees WHERE user_id IS NULL;

-- Users without employee match  
SELECT * FROM users WHERE id NOT IN (SELECT user_id FROM employees WHERE user_id IS NOT NULL);

-- Duplicate user assignments
SELECT user_id, COUNT(*) FROM employees WHERE user_id IS NOT NULL GROUP BY user_id HAVING COUNT(*) > 1;
```

## Rollback Plan
- Migration down() removes user_id column
- No data loss (column is nullable, all data remains)
- Relationships become implicit (email+school matching) again
