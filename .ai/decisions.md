## [2026-04-26] — Adopt ai-workspace session protocol
**Decision**: Layer the `ai-workspace` AI session protocol on top of Laravel Boost — relocate Boost guidelines to `.ai/laravel-boost.md`, make `CLAUDE.md`/`AGENTS.md`/`GEMINI.md`/`AGENT.md` thin pointers to `AGENTS.md` (canonical), seed `.ai/*` markdown set, link `~/SecondBrain`.
**Reason**: ~70% per-session token reduction (scoped reads vs auto-loading 309-line Boost dump every session) and persistent cross-project memory via SecondBrain evolution loop.
**Trade-off**: Re-running `php artisan boost:install` will overwrite the pointer files — mitigated by `scripts/restore-pointers.sh`. Slight cognitive overhead remembering "AGENTS.md is canonical, not CLAUDE.md."
**Revisit if**: Boost regeneration becomes routine (>1×/month) and the restore script gets skipped, OR if a future tool stops respecting `AGENTS.md` as the single source.

## [2026-04-27] — DB-agnostic stance (PostgreSQL + MySQL)
**Decision**: Project must work on both PostgreSQL 14+ and MySQL 8.0+. Use Laravel's Schema builder for all column types and migrations; no raw driver-specific SQL except behind a `getDriverName()` guard with both branches implemented. JSON columns use `$table->json(...)`, never `text` for JSON payloads.
**Reason**: Discovered drift — `.ai/memory.md` documented MySQL but `.env` runs pgsql. Filament's `data->>'format'` notification query failed because `notifications.data` was `text` (works on MySQL, not Postgres). DB-agnostic posture prevents this class of bug at the source.
**Trade-off**: Cannot use Postgres-only features (partial indexes via raw SQL, JSONB-specific operators, generated columns) without dual implementations. The earlier-proposed `WHERE returned_at IS NULL` partial unique index for equipment assignments must therefore use a model observer + transaction lock instead, or a dual-driver migration.
**Revisit if**: One driver becomes the explicit production target and the other can be dropped, OR a critical feature requires Postgres-only capabilities (full-text search, JSONB GIN indexes).

## [2026-04-27] — Unify Identity (Employee canonical, User auth-only)
**Decision**: `Employee` holds all personal/organizational data (name, school, division, employment). `User` holds only auth concerns (email, password, approval_status, google_id, roles). Link is `employees.user_id` — unique, nullable, `nullOnDelete`. `User::name`/`school_id`/`division_id`/`school` are accessor-delegated through the employee relation; existing callsites (SchoolScope, Filament resources, middleware) keep working unchanged.
**Reason**: Two tables stored the same person twice; sync drift was a real, recurring problem (the now-deleted `scripts/sync_employees_to_users.php` was evidence). DRY single source of truth. Bulk HR import no longer creates spurious login accounts.
**Trade-off**: Reads of `$user->school_id` lazy-load the employee relation (acceptable — Eloquent caches per instance). Pure-auth users (super-admin) have `employee_id = NULL` and `name` falls back to email. UserResource form picks an Employee instead of editing name/school directly — different UX, but reflects reality.
**Revisit if**: A future requirement needs users that are entirely decoupled from human identity (machine accounts, service tokens). Likely solved by a separate `service_accounts` table rather than re-coupling.

## [2026-04-28] — Field-Ops PWA + scoped REST API
**Decision**: Build a Nuxt 3 PWA in `pwa/` subdir backed by a new Laravel REST API under `routes/api.php`. Sanctum tokens (one per device, 30d expiry). The API exists *only* for PWA field flows: equipment return (with photo→PDF proof), new equipment issuance, QR equipment lookup, and ticket creation with attachments. Filament remains the system-of-record for all admin operations. Full plan in `~/.claude/plans/i-am-planning-to-zany-reddy.md`.
**Reason**: Field staff need offline-capable mobile workflows (offline = hard requirement, forces a real frontend). Decision 007's "no API" stance was correct for admin tooling but wrong for field ops.
**Trade-off**: Adds an entire frontend stack + auth layer + ~6 weeks of work. Two surfaces to keep in sync via versioned API contracts. Reverses 007 partially.
**Revisit if**: PWA scope creeps into admin operations (re-merge into Filament), or a native wrapper becomes mandatory (then PWA is the wrong target).

## [2026-04-28] — Local dev only — no Docker/Sail
**Decision**: Stop using Laravel Sail and Docker for local development. Run `php artisan ...`, `composer ...`, `npm ...`, `psql ...` directly against host-installed PHP 8.4 and PostgreSQL. `docker-compose.yml` and `docker/` stay in repo for production deploy reference but are not invoked in dev. Supersedes Decision 006.
**Reason**: User is CPU-constrained on the dev machine; Docker overhead is non-trivial and unnecessary for a single-dev local setup.
**Trade-off**: Loses Sail's environment parity guarantee — local-only dev means subtle "works on my machine" risk for Postgres extensions, PHP version mismatches, etc. Mitigated by the DB-agnostic stance (Decision 010) and CI (when added).
**Revisit if**: A second dev joins the project, OR a production-only dependency (e.g., Redis cluster, S3) becomes hard to mock locally.

## [2026-04-27] — Drop drifty resource README files
**Decision**: Removed `EmployeeResource/README.md`, `EquipmentResource/README.md`, `SchoolResource/README.md`, `TicketResource/README.md`. Kept only `AssignmentResource/README.md` (recently rewritten and accurate).
**Reason**: All four documented APIs and field names that did not exist in code. The same drift that bit the AssignmentResource README. Fixing them takes ongoing effort that nobody pays. Canonical info lives in `.ai/memory.md` (Domain Rules + Modules), the models themselves, and `.ai/context.json`.
**Trade-off**: Loss of per-resource onboarding doc. Worth it: a wrong doc is worse than no doc.
**Revisit if**: A resource grows complex enough to warrant a dedicated README (e.g., AssignmentResource, which has a service-class contract worth documenting), at which point treat that single README as load-bearing and review it on every PR that touches the resource.
