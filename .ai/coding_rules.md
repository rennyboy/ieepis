# Coding Rules & Conventions (IEEPIS)

> Style and AI-interaction rules. Framework-specific Laravel/Filament rules live in `.ai/laravel-boost.md` (load on demand).

## Memory File Rules
- `.ai/*` files stay under 50 lines each — except `laravel-boost.md` (on-demand reference, allowed to be large)
- `handoff.md` max 25 lines; rotate excess to `SecondBrain/Daily Notes/YYYY-MM-DD.md`
- `decisions.md` entries always include: **Decision / Reason / Trade-off / Revisit if**
- `tasks.md` uses checkbox `[ ]` / `[x]` with sprint sections (P1/P2/P3)

## AI Interaction Rules
- Always load `.ai/handoff.md` and the `.ai/memory.md` index before any task
- One role per session — don't mix architect and debugger in the same chat
- Update `.ai/handoff.md` before ending a session with meaningful work
- Scope prompts narrowly: *"Read `.ai/memory.md` Domain Rules section. Then propose Y."*
- Never re-paste project context in chat — reference it
- For framework code, also load the relevant section of `.ai/laravel-boost.md`

## PHP / Laravel Conventions (summary — full set in `.ai/laravel-boost.md`)
- PHP 8.4 features: constructor promotion, enums, named args, readonly, explicit return types
- Curly braces on every control structure, even single-line
- PHPDoc blocks over inline comments
- TitleCase Enum keys: `GoodCondition`, `ForDisposal`
- No business logic in controllers — use service classes or model methods
- All `make:` commands via `vendor/bin/sail artisan make:...`
- Run Pint before commit: `vendor/bin/sail bin pint --dirty --format agent`

## Filament Conventions (summary — full set in `.ai/laravel-boost.md`)
- `static::make()` pattern for component initialization
- Resources in `app/Filament/Resources/` — one per model
- Pages auto-generate in `app/Filament/Resources/{Model}Resource/Pages/`
- Use `relationship()` on Select/Checkbox when linking related models
- Implement `canViewAny()` / `canCreate()` / `canEdit()` on every Resource

## Domain Invariants (do not break — see `.ai/memory.md` Domain Rules)
- No `employee_id` FK on `equipment` — accountability via `equipment_assignments` only
- One active assignment per equipment — enforced in model
- Soft deletes on all core models — use `->withTrashed()` when needed
- QR codes auto-generated by Observer — don't duplicate
- `school_admin` global scope — never bypass without explicit reason

## Vault Rules (`SecondBrain/`)
- One note per concept — no mega-notes
- Daily notes prefixed `YYYY-MM-DD `
- Decision notes in `Decisions/`, not inline in project notes
- Prompts in `Prompts/`: context → task → output format

## General Style
- Markdown only — no HTML in notes
- `##` headers, never bold-as-section
- Bullets over paragraphs for scannability
- No trailing commentary — notes are reference, not narrative
